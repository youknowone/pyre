# majit Compatibility Matrix

Last updated: March 13, 2026

This file is the Phase 0 seed artifact for tracking equivalence with the in-tree RPython JIT sources.

## Legend

- `implemented`: intended to work and covered by focused tests
- `partial`: some behavior exists, but parity is incomplete
- `stubbed`: code path exists, but semantics are placeholder or incomplete
- `unsupported`: not implemented and should fail loudly if reached

## Backend Policy

The Cranelift backend must obey the following rules:

1. Unsupported opcodes must return `BackendError::Unsupported`.
2. Void opcodes are not allowed to silently degrade into no-ops unless they are explicitly documented as semantic no-ops.
3. Guard exits and `FINISH` exits must preserve `Int`, `Ref`, and `Float` values exactly.

## Immediate Gaps

These are the highest-priority items currently blocking equivalence:

- Guard-bearing `CALL_ASSEMBLER_*` and recursive call dispatch are incomplete (only finish-only callee loops work).
- Only the narrow single-call + immediate-guard `CALL_MAY_FORCE_*` / `GUARD_NOT_FORCED` subset is supported; general savedata/resume paths and broader jitframe state are still missing.
- Full jitframe reconstruction from arbitrary caller stack frames is not yet implemented; `EncodedResumeData` now roundtrips compact fail-arg numbering plus pending writes, compiled traces preserve terminal `FINISH`/`JUMP` exit layouts, but the backend still does not emit complete frame layout metadata for all guard sites and caller frames.
- Card marking and incremental GC semantics are incomplete.
- Driver macros still lack full RPython-equivalent declarative lifecycle validation; the supported `#[jit_interp]` subset now rewrites marker macros into real driver calls, accepts optional marker-local green tuples on explicit marker forms, rejects inconsistent binding names or green tuples, rejects `can_enter_jit!` before the first `jit_merge_point!`, and preserves configured `greens = [...]` as structured `GreenKey` tuples on the runtime driver path. `#[jit_driver]` now validates green/red overlap plus optional virtualizable metadata, requires that the virtualizable red be `Ref`-typed when runtime descriptor types are provided, implements a runtime-facing `DeclarativeJitDriver` trait, and runtime `JitDriver` instances can attach that declarative descriptor to `TraceCtx` or build directly from the declarative driver type. The runtime now also rejects declarative red/live type mismatches up front, preserves typed `Ref`/`Float` reds through trace inputargs plus a narrow typed compiled-entry/exit path, routes declarative virtualizable metadata through explicit `JitState` lifecycle hooks even when only the declarative name is known, lets `VirtualizableInfo` enrich the same seam when configured, has an opt-in automatic heap-sync path for virtualizable boxes, and can now drive a narrow blackhole-fallback restore path through `JitState::restore_guard_failure`, including automatic typed reconstructed-frame projection through the new per-frame hooks, common pending field/array write replay through `PendingFieldWriteLayout` even for virtual targets/values, multi-frame resume hooks plus fallback custom stack restoration, preserved nested virtual graphs plus cached repeated virtual materialization through `materialize_virtual_ref_with_refs`, with cache reuse into pending-write replay on the common generic cached restore path, but full `JitDriver` parity is not finished.
- `#[jit_inline]` exists only for simple integer helpers; bare `calls = { helper }` entries can now infer helper policy from `#[jit_inline]` / `#[elidable]` / `#[dont_look_inside]`, `#[jit_interp(auto_calls = true)]` can now infer direct top-level helper calls without a call map, and nested `#[jit_inline]` helpers can now infer direct helper calls without a local map. The current integer-helper call subset is tested through arity `0..=8`, inferred/auto annotated helper calls can now normalize mixed integer-like argument signatures plus non-`i64` integer-like returns such as `u32`, `usize`, and `bool` through hidden wrappers, and annotated helpers can now use `residual_void_wrapped`, `residual_int_wrapped`, and `elidable_int_wrapped` to get the same wrapper-backed behavior through explicit call maps, including from nested `#[jit_inline(calls = { ... })]` helpers. Fully automatic whole-module helper discovery, ref/float helper signatures, and raw explicit-call ABI generalization are still missing.
- Virtual ref finishing and full FORCE_TOKEN/GUARD_NOT_FORCED lifecycle need completion.
